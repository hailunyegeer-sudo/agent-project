# Agent开发实践指南

## 文档目的

本文件用于测试 `agent开发` 这种中英混合、紧密相连的查询。标题与正文中连续出现“Agent开发”，用于确认检索器不会只把查询拆成 `agent` 和 `开发` 后进行松散匹配，而是能够把完整主题作为优先条件。

## Agent开发的最小闭环

Agent开发的最小闭环包括四部分：接收用户任务、构造上下文、调用模型决策、根据需要执行工具并返回最终答案。对于本地工具型 Agent，B1 负责运行与消息管理，B2 提供技能函数，B3 负责工具说明和工具调用，B4 负责 LLM 决策。B5 则读取或保存记忆，使后续任务能够复用历史上下文。

## 设计原则

在 Agent开发中，首先应明确各模块边界。B2 不决定是否调用工具，也不维护 messages；B4 不执行工具，只输出 AIMessage；B3 不替模型做选择，只校验和执行已有 tool_calls；B1 不直接实现业务 Skill，而是将模型输出和工具结果串成一个循环。边界清晰以后，调试时才能明确判断问题来自模型、schema、参数、Skill 还是消息管理。

## 验证场景

检索 `agent开发` 时，本文应属于精确主题文档：`phrase_matched` 应为 `true`。即使其他文件分别出现“Agent”和“开发”，只要不连续表达 Agent开发 这一主题，排序也不应超过本文。这个案例专门用于验证“完整查询短语 + 辅助词召回”的组合策略。

## 后续扩展

Agent开发进入进阶阶段后，可以继续加入复杂检索、复合 Skill、错误码体系、工具缓存、重试策略、Plan-and-Execute 流程以及不同 schema 描述对工具调用准确率的影响实验。无论扩展到什么程度，基础分层与可观察日志都应保留。
